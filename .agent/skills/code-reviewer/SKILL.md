# Code Reviewer (代码审阅专家)

本 Skill 用于标准化处理项目的代码审阅任务，特别是针对特定文件的深度检查与反馈。

## 逐文件审阅规范 (File-by-File Review Guidelines)

当被要求审阅多个文件时，必须严格遵守以下流程：
1. **单独任务处理**：对于给出的每个文件，依次处理，每个文件用一个单独任务（或轮次）解决，不要将所有文件混在一起。
2. **核心动作**：对于每个文件，必须完成以下三项工作：
   - 讲解代码：简明扼要地解释该文件的作用及核心逻辑。
   - 注释补充：直接修改该文件以补充详尽的代码注释（Docstrings和关键行内注释）。
   - 审阅修改内容：评估开发人员所做的改动是否合理，是否符合架构规范。
      - **结论表达限制**：避免浮夸的修辞，直接陈述结论。
      - **客观评价限制**：必须同时列举改动的“优点”、“缺点”和“漏洞/风险”。
      - **强制编号限制**：每个列举出的优点、缺点和漏洞/风险都必须分配一个统一格式的唯一 ID 编号（例如：`[REV-001]`），以便后续引用和追踪。
      - **初学者友好与实操性限制**：在指出“缺点”和“漏洞/风险”时，**必须定位到具体的代码行号位置**，**必须使用Python初学者能看懂的大白话和生活比喻来解释问题**，并且**必须直接提供一份可行的修正方案或重构思路**。
3. **红线限制**：**禁止修改代码的业务逻辑本身！** 仅允许添加、修改注释。注意严格区分代码逻辑和注释的区别。

## 问题报告与文档记录规范

如果在审阅过程中发现代码存在任何问题（Bug、架构缺陷、命名不规范等）：
1. **先询问确认**：必须**直接告诉用户**发现的问题，等待用户决定是否记录以及记录到哪个层级（P0/P1/P2/长期任务）。
2. **文档维护位置**：所有的审阅结论和建议必须记录在 `docs/AI审阅` 目录下的 Markdown 文件中。
   - **强制包含审阅进度**：在审阅报告的顶部，必须建立一个叫 `## 审阅进度列表` 的章节。使用 Markdown 的待办事项格式（`- [ ]` / `- [x]`）来完整列举出本次提交关联的所有被修改或新增的文件，并随着多轮次文件审阅的推进过程，动态为你刚刚检查完毕的文件打上已完成的 `[x]` 标记。
3. **文件名映射**：
   - 每一个对应的开发规格（Spec，通常位于 `.trae/specs/` 下）对应一个专属的审阅报告。例如 `implement-p0-t2-skeleton-protocol_审阅报告.md`。
   - 如果是长期或待定任务，记录在单独的 `docs/AI审阅/长期待办任务追踪.md` 中。
4. **文件内容结构**：
   - 使用固定的章节名表示优先级：`## P0 级建议与结论（核心阻断/架构级调整）`，`## P1 级建议与结论（重要功能/体验优化）`，`## P2 级建议与结论（边缘优化/代码规范）`。
   - 每个章节下使用无序列表 `- [ ]` 记录具体的审阅建议和结论。
