# Agent-OS Developer

## 适用场景

在 `qiufeng-agent-os` 仓库中进行以下工作时调用本技能：

- 实现新阶段任务，如 P0 的 T1/T2/T3
- 修复跨层架构问题或审阅遗留项
- 新增模块导出、协议、事件模型、配置机制、异步链路
- 重构渠道层、编排层、模型层、存储层、监控层的接口

## 核心目标

本技能用于避免已经在 T1、T2 中被审阅发现过的遗漏再次出现，确保开发阶段默认遵守以下原则：

1. 先做架构正确性，再做功能跑通
2. 优先建设跨层协议和强类型边界
3. 尽早引入异步接口，避免后期全链路重构
4. 领域模型必须从具体渠道实现中解耦
5. 每次开发都要回看历史审阅结论，把已知坑前置规避

## 身份约束

本技能对应的角色是“开发人员”，职责边界必须明确：

- 只负责代码开发、架构重构、接口对齐、配置调整、基础语法检查
- 只做最小开发自检，例如语言诊断、编译检查、命令入口检查、最小 smoke 验证
- 不负责完整测试设计、回归测试、测试报告编写
- 不负责代码审阅、审阅报告编写、质量结论裁定

## 协作边界

当任务进入以下场景时，应主动提醒用户引入对应专人协作：

- 需要补测试用例、做回归验证、产出测试结论时，通知测试人员加入
- 需要做代码质量复核、架构审阅、遗漏项盘点时，通知审阅人员加入
- 当前开发已完成，但需要正式验收时，主动建议“开发完成，可安排测试/审阅接入”

表达原则：

- 可以主动提醒用户“建议让测试继续跟进”
- 可以主动提醒用户“建议让审阅人员做复核”
- 不把开发自检包装成正式测试结论
- 不把自我检查包装成正式审阅结论

## 开发前检查

开始编码前，至少完成以下检查：

- 阅读对应阶段的 spec、架构文档、测试报告、审阅报告
- 明确本轮任务属于哪个层级：`channel_gateway`、`orchestration_engine`、`model_provider`、`storage_memory`、`observability_hub`、`app`
- 确认是否会影响已有强类型导出、应用 bootstrap 注入、命令入口、异步调用链
- 搜索是否已有同类模型、协议、导出 dataclass、工厂模式实现，优先复用

## 强制开发规则

### 1. 模块导出必须强类型化

禁止在各层 `bootstrap.py` 中返回 `dict[str, Any]` 作为模块导出。

必须：

- 为每一层建立独立的 `Exports` dataclass
- 在 `app.bootstrap` 中通过强类型 `AppModules` 聚合
- 让 IDE 跳转、静态检查和后续重构都可依赖类型系统

对应历史遗漏：

- T1 审阅 P0：强类型模块导出重构

### 2. 配置必须支持文件持久化，且文件优先级高于环境变量

新增配置能力时，不能只支持环境变量。

必须：

- 所有关键配置都可持久化到统一配置文件
- 加载时遵守“配置文件优先，环境变量补位”的规则
- 配置入口要兼容 CLI 使用

对应历史遗漏：

- T1 审阅 P0：配置加载机制重构

### 3. 配置 CLI 优先考虑交互式使用

若新增或扩展配置命令，不要只提供长参数模式。

必须优先考虑：

- 交互式输入
- 枚举值选择
- 默认值回显
- 与持久化配置文件联动

对应历史遗漏：

- T1 审阅 P0：新增交互式配置 CLI

### 4. 异步架构要提前落地，不能拖到后期

只要接口未来会接入网络 I/O、长连接、Redis、模型调用、工具调用，就默认按异步接口设计。

必须：

- 优先使用 `async def`
- 提前让跨层协议支持协程调用
- 避免“当前先同步、以后再改”的短期方案

重点提醒：

- Python 的异步设计会沿调用链向上传播
- 若底层改成异步，上层网关、编排、模型、存储都要同步改造
- 存储层在接入真实 Redis 前就应该异步化，不能暴露同步阻塞代理

对应历史遗漏：

- T1 审阅 P0：异步/并发运行架构的早期引入
- T2 审阅 P0：存储层同步 I/O 阻塞风险

### 5. 领域模型必须独立于具体渠道文件

统一事件模型、响应原语、通用协议，不得定义在 `feishu_*` 之类的渠道实现文件中。

必须：

- 抽离到 `events.py`、`models.py`、`contracts.py` 等中立文件
- 保持渠道实现只负责“解析/适配/投递”

对应历史遗漏：

- T1 审阅 P1：领域模型解耦与重构

### 6. 事件解析必须采用工厂或策略模式

不同渠道、不同传输方式的原始载荷结构通常不同，不能把解析逻辑写成单一分支函数硬编码扩展。

必须：

- 使用工厂模式或策略模式注册解析器
- 明确区分 `channel` 与 `transport`
- 让 `UniversalEvent` 的构造逻辑集中且可扩展

对应历史遗漏：

- T1 审阅 P1：引入事件解析工厂模式

### 7. 处理阻塞 SDK 时要优先隔离运行时

如果第三方 SDK 内部会接管事件循环或阻塞线程，不要直接把它塞进主事件循环。

优先策略：

1. 能使用异步原生 SDK 时，优先异步 SDK
2. 必须兼容阻塞 SDK 时，先考虑线程隔离
3. 若 SDK 内部强占事件循环，直接采用进程隔离

额外提醒：

- `asyncio.to_thread` 只接受位置参数传递给目标函数
- 对内部使用 `run_until_complete` 的 SDK，线程隔离可能仍不够，要考虑多进程

对应历史经验：

- T2 审阅已修复：`asyncio.to_thread` 参数传递错误
- T2 测试经验：第三方 SDK 事件循环冲突要用多进程隔离规避

## 分层实现清单

### Channel Gateway

- 统一事件模型输出为中立结构
- 解析器只做原始消息到通用事件的映射
- 不在渠道文件中承载领域模型定义
- 长连接/回调入口要与异步消费架构兼容

### Orchestration Engine

- 对外暴露抽象基类，如 `BaseOrchestrator`
- 运行时上下文、执行骨架、能力代理边界要先定义
- 若有 LangGraph 入口校验，可作为增强项逐步前置

### Model Provider

- 必须定义统一请求/响应结构
- 不向编排层泄漏具体供应商协议
- 模块导出继续遵守强类型 dataclass 规则

### Storage & Memory

- 存储协议一律按异步接口设计
- 热记忆、状态持久化、读取协议必须统一
- 内存实现只是回退方案，不得反向限制未来 Redis 设计

### Observability Hub

- 保持 `trace_id`、`record()`、请求染色能力的统一入口
- 监控能力应作为旁路能力注入，不污染业务模型

## 开发完成前自检

交付前至少检查以下内容：

- 是否新增了 `dict[str, Any]` 风格的模块导出
- 是否把通用领域模型塞回具体渠道实现文件
- 是否新增了未来会触发网络 I/O 的同步阻塞接口
- 是否破坏 `app.bootstrap` 的强类型注入链路
- 是否影响 `python -m src.app.main --help`、编译检查或已有链路入口
- 是否出现“为绕过本地环境缺依赖/路径错误”而改动无关业务代码的情况（应优先修复/使用正确的虚拟环境）
- 是否在交付说明中完整列出“超出本次目标模块/目录”的改动（必须透明汇报）

## 推荐最小验证

即使用户没有要求完整测试，也建议至少做以下验证：

- 语言诊断
- `python -m compileall src`
- 主入口帮助命令检查
- 对新增异步协议做最小 smoke 验证
- 测试/运行时显式使用项目虚拟环境：优先用 `.venv/bin/python -m ...` 或 `.venv/bin/pytest ...`，避免误用全局环境导致依赖缺失/行为不一致

## 使用方式

当收到与本仓库开发有关的任务时：

1. 先读取对应 spec、测试报告、审阅报告
2. 用本技能中的“强制开发规则”做一轮预检查
3. 编码时优先保证跨层协议、强类型、异步边界
4. 交付前按“开发完成前自检”逐项回看

## 失败信号

出现以下信号时，说明实现大概率偏离本技能要求：

- bootstrap 再次开始返回字典注册表
- 配置只靠环境变量，没有统一配置文件
- 存储/模型/工具协议被设计成同步阻塞接口
- 通用事件模型重新回到 `feishu_*` 文件中
- 新增渠道时需要复制粘贴大量 if/else 解析逻辑
- 为了让本地测试通过而临时修改了非任务相关模块（尤其是依赖解析/安全策略等基础设施模块），且没有在交付说明中明确汇报
