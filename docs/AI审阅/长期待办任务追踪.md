# 长期待测与待实现任务追踪

## 待测试任务 (Pending Tests)
* [ ] **飞书 Webhook 模式连通性测试**：目前仅测试了长连接模式。由于本地开发环境缺乏公网 IP，Webhook 模式（基于 `verify_token` 和 `encrypt_key` 的验签与解密）尚未进行真实环境端到端验证。需在后续有公网映射（如 ngrok）或测试服部署时补充验证。

## 待实现任务 (Pending Implementations)
* [ ] **全量交互式配置 CLI**：基于终端交互库（如 `questionary` 或 `InquirerPy`），结合 `rich` 库，实现一个菜单式的系统配置引导程序，并支持将所有配置序列化到统一的本地配置文件中。
* [ ] **飞书 API 客户端升级**：随着业务深入，如果运行期需要高频调用飞书 API（如发送富文本消息、拉取文档），建议将目前基于 `urllib` 的轻量级实现重构，引入 `aiohttp` 或飞书官方 `lark-oapi` 库，以支持连接池复用和更复杂的异步请求。
* [ ] **补齐第三方库的类型存根 (Type Stubs)**：当前依赖的 `nonebot2` 和 `lark_oapi` 缺少官方的 py.typed 标记和类型存根文件，导致 `mypy` 无法进行深度静态分析。未来在进行严格代码审计前，需考虑自行编写关键接口的 `.pyi` 文件，或等待官方更新。
* [ ] **运行时上下文深拷贝优化**：`src/orchestration_engine/runtime_context.py` 中的 `snapshot()` 方法目前使用 `dict()` 进行浅拷贝。如果未来 `state` 中存储了复杂的嵌套对象（如自定义的 Pydantic 模型实例），当前的快照机制可能无法防范深层数据的引用篡改，导致状态回滚失败。建议在业务变复杂时引入 `copy.deepcopy` 或序列化机制来保证快照的绝对隔离。
* [ ] **InMemoryStore 线程安全与 OOM 防范**：`src/storage_memory/contracts.py` 中的 `InMemoryHotMemoryStore` 缺失锁机制，在多线程环境（如多线程 Webhook 处理请求）下直接操作字典会导致竞态条件。此外，`ltrim` 方法的边界条件（当 `stop == -1` 且列表为空时）可能无法正确收敛，且未对内存容量做全局限制，高频请求下存在内存泄漏（OOM）风险。
* [ ] **Lambda 类型推断修复**：在 `src/storage_memory/bootstrap.py` 等模块中使用了 `lambda` 闭包，导致静态检查工具 `mypy` 无法推断其复杂签名（提示 `Cannot infer type of lambda`）。未来应重构为具名包装函数。
* [ ] **请求染色判定机制优化**：目前 `src/observability_hub/request_coloring.py` 中的判定逻辑存在两个待优化点：1. 状态内聚性不足，调用 `is_request_colored` 必须主动传入 `state`，未来应考虑结合 `contextvars` 隐式传递；2. 提取逻辑风险，`_read_value` 目前仅支持字典浅层键值判断，如果上游传入深度嵌套的上下文（如 `{"header": {"traceId": "..."}}`），会导致漏判，需引入类似 JSONPath 的深层提取机制。
* [ ] **编排器返回值强类型重构**：`src/orchestration_engine/base_orchestrator.py` 中的 `BaseOrchestrator.execute` 方法目前返回 `dict[str, object]`，这种返回值类型约束比较弱，虽然提供了灵活性，但也丢失了强类型带来的安全感。后续编排引擎执行结果如果结构固定，应考虑抽象为专门的强类型数据类。
* [ ] **模型路由逻辑解耦重构**：`src/model_provider/bootstrap.py` 中的 `RoutedModelProviderClient.invoke` 方法使用了硬编码的 `if-else` 分支（如 `if is_minimax_request(request):`）。随着未来新增 OpenAI、Anthropic 等更多供应商，这种写法会导致方法无限膨胀，应重构为策略模式或基于配置的注册表模式。
* [ ] **模型请求状态透传的深拷贝保护**：在 `src/model_provider/litellm_adapter.py` 的 `build_litellm_completion_payload` 方法中，处理 `metadata` 时仅使用了浅拷贝（`dict(request.metadata)`）。如果上层传递了嵌套层级较深的可变对象（如列表、字典），在后续修改或异步透传时可能会篡改原始请求数据，需引入深拷贝机制保障状态隔离。
* [ ] **全局异常捕获与隔离兜底**：在 `src/skill_hub/capability_hub.py` 的 `RegisteredCapabilityHub.invoke` 方法中，执行 `await handler(request)` 时缺少 `try...except` 块包裹。如果下游工具抛出未处理的异常，会导致调用链崩溃，破坏编排层的并发稳定性，需补充统一的能力调用异常拦截机制。
* [ ] **反向注册机制重构**：`ModelCapabilityRouter` 向 `RegisteredCapabilityHub` 注册时需要显式调用 `register_into` 传入容器本身（反向依赖）。后续如果扩展出更多复杂能力组，会导致注册代码变冗长。可以考虑引入依赖注入框架或使用基于装饰器的自动发现机制。
