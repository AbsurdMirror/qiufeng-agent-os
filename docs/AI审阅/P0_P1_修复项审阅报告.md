## P1 级建议与结论（重要功能/体验优化）

### 1. `src/channel_gateway/feishu_sender.py` 及 `responses.py` (飞书长文本分片发送)
- **优点**：去除了硬性的长度拦截，改为按最大允许长度对文本切片（分块发送）。并在发送时，只对首个分片保留 `reply_to` 关系，后续片段按普通消息顺序追加，有效地解决了大模型长回复在渠道端丢失的体验痛点。同时引入了分块发送过程中的部分失败处理。
- **缺点**：按字符串长度硬切分，缺乏对内容语意的识别。
- **漏洞/风险**：由于大模型常输出 Markdown 格式的文本或带 Emoji 等双字节特殊字符，简单按索引 `[i:i+max_chunk_size]` 进行切片可能会截断 Markdown 语法（如代码块的三反引号 `` ``` `` 被劈开）或导致字符乱码，进而在飞书客户端上渲染失败。另外，若第 N 片发送失败中断，前 N-1 片已发出，会导致向用户传达不完整的截断信息。

- [ ] **待定/长期优化任务**：在分片发送逻辑中引入智能切片策略（如按段落、Markdown 代码块边界进行软切分），或引入消息重发补偿机制以避免发出一半的截断现象。
