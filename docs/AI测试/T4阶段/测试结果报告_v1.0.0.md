1. **身份去重控制器 (AT-01 / AT-02)**：
   - 实例化 `SessionContextController`，利用 UUID 解析逻辑严格保证了 `get_logical_uuid` 返回的一致性。

3. **内存级别 LIFO 与全量持久化 (AT-04 / AT-05)**：
   - 启用抽象的 `InMemoryHotMemoryStore` 作为基础 Carrier 测试。

### 3.3 异常记录与修复动作（自动化测试阶段）
- **ToolParser 默认值解析缺陷**：在初始阶段发生了一次断言错误。原因在于定义函数时，Pydantic `Field` 中的 `default=10` 若未与 Python 函数本身的缺省赋值 `=10` 同步绑定，会导致系统把该字段误识为必填（Required）。目前该行为已通过测试修正约束，并成功映射正确的 JSON Schema `"default": 10`。
- **并发环境就绪**：测试代码成功接纳了 `pytest-asyncio` 的修饰挂载，证明基于协程运行期的代码设计对于第三方堵塞非常免疫并具有优秀的跨架构兼容性。

### 4.1 调试过程中发现的 Bug 及修复

| 序号 | Bug 描述 | 根本原因 | 修复方案 |
|---|---|---|---|
| **BUG-01** | 模型调用直接路由到 `InMemoryModelProviderClient`（mock），返回内容是用户原话回显 | `model_tag="model.minimax.chat"` 与 `ModelRouter.clients` 字典中的 key `"minimax"` 不匹配，回落到 `"default"` | 改为 `model_tag="minimax"` |
| **BUG-02** | 工具执行后发送第二次请求报 MiniMax 2013 错误：`tool result's tool id() not found` | `ModelMessage` 只有 `role/content` 字段，转换时丢失了 `tool_calls` 和 `tool_call_id`，MiniMax 无法匹配工具调用 | 第二次请求通过 `litellm_kwargs["messages"]` 直接覆盖，完整保留 `tool_call_id` / `tool_calls` 结构 |
| **BUG-03** | 重启后历史记忆始终为 0 条，无法续联 | `SessionContextController` 的 `open_id → UUID` 映射仅存内存，每次重启生成新 UUID，Redis key 不匹配 | 测试脚本中 `session_id` 直接使用稳定的 `event.user_id`（飞书 `open_id`）绕过不稳定的 UUID 映射 |

> **架构待改进项**（已在代码注释中标记）：`SessionContextController` 的 UUID 映射关系本身也应持久化到 Redis，以彻底解决 BUG-03 的根因，而非在测试层绕过。
