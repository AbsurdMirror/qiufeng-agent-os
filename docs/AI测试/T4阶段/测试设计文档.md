# Agent-OS T4 阶段测试设计文档

## 1. 测试目标与范围

**目标**：本次测试聚焦于验证 Agent-OS 在 T4 阶段（数据流转与上下文管理）全新引入的核心业务链路与状态贯通机制。重点考察底层热记忆缓冲、防重处理、ID 会话映射以及针对大语言模型调用的格式自动修正能力。同时，兼顾 T1-T3 已打通的基础功能的回归兜底。

**测试范围梳理**：
1. **身份与流量网关**：`channel_gateway.session_context.SessionContextController` -> 验证 `open_id` 向逻辑 `UUID` 的归一映射机制，及并发场景下基于滑动窗口的消息去重。
2. **工具格式解析底座**：`skill_hub.tool_parser` -> 验证从 Pydantic V2 模型零损失翻译为 OpenAI 标准 JSON Schema 的强校验与映射逻辑。
3. **记忆存储与持久化调度**：`storage_memory.redis_store.RedisHotMemoryStore` -> 验证 Redis 如何依托 LIFO（后进先出 / 淘汰旧数据）策略保证短期高频对话记忆的高效写入与精确防溢出。
4. **端到端多进程 / 异步运行架构**：通过人工终端发送测试请求，检验在 T4 环境下并发异步（Event Loop）不被框架或长连阻塞的最终成效。

## 2. 自动化测试方案 (Automated Tests)

| 测试项 ID | 测试模块与场景 | 前置状态 | 数据输入设计 | 程序预期行为与验收标准 | 对应实现 Spec |
| --- | --- | --- | --- | --- | --- |
| **AT-01** | **身份映射幂等性**<br/>验证 `get_logical_uuid` | `SessionContextController` 常规初始化完毕 | 连续两次传入相同的物理渠道 ID：`open_id="ou_wx123"` | - 两次输出的 `uuid` 必须完全相等且合法<br/>- 传入另一ID `ou_wx456` 时需生成不同的 `uuid` | GW-P0-05 |
| **AT-02** | **滑动窗口防重过滤机制**<br/>验证 `is_duplicate` | 给定防重时间窗口 = `100ms`（仅为快速测试设置） | 第一次输入 `msg_id="UUID_1"`<br/>隔 `10ms` 再次输入 `UUID_1`<br/>等待 `150ms` 后再输入 `UUID_1` | - 第一次返回 `False`（已放行）<br/>- 第二次返回 `True`（被拦截）<br/>- 第三次返回 `False`（窗口超期记录被清理，可在此放行） | GW-P0-06 |

## 3. 端到端综合手动测试方案 (Simple Agent 联调测试)

**设计思路说明**：
为了最真实地验证 T4 阶段新增的功能（工具调用解析、历史对话追溯、状态全量上下文存储与重启），我们将不采用零散的单一断点测试。而是**整体实现为一个基础的智能体 (Simple Agent)**，为其挂载真实的飞书长连接、真实的 MiniMax 模型底座，并配备几个专用于测试的原子工具。
接着，通过规划好的**连贯对话剧本**，由测试人员在飞书客户端上“跑完全程”，从而实现所有特性的串联闭环验证。

### 3.1 前期系统环境准备 (Agent Setup)
您或您的开发团队需要执行以下脚本/环境搭建（在提供辅助运行时代码的前提下）：
1. **渠道与模型接入**：在 `.env` 中正确配置 `FEISHU_APP_ID`, `FEISHU_APP_SECRET` 以及 MiniMax 的 `API_KEY`。确保 Redis 实例在后台正常运行（极为关键，这是状态流转的核心）。
2. **测试脚本与辅助工具封装**：**请不要污染 `src` 源码目录！** 所有的辅助测试逻辑应当收敛在 `tests/T4/scripts/run_manual_test.py` 中。您可以在该测试脚本内部，临时定义两到三个附带清晰 Doxygen + Annotated 标注的 Dummy Tools，利用系统的注册接口动态挂载给测试专用 Agent。例如：
   - `get_weather(city: str)`：无论输入什么都返回固定文本“天气晴朗，25度”。
   - `calculate_sum(a: int, b: int)`：返回两数之和。
3. **服务启动挂起**：直接通过 `python -m tests.T4.scripts.run_manual_test` 启动该测试桩实例，将编排引擎图调度与上述组件接通。在终端保持进程挂起（阻塞等待），直到您的终端与飞书应用端均显示 "已连接"。

---

### 3.2 飞书对话测试剧本 (Test Scenarios Flow)
测试人员请在飞书对话框内，严格按照以下回合（Rounds）推进测试，期间注意同时观察后端的控制台日志流向。

#### 🟢 回合 1：基础连接与身份锚定 (测试 T1/T4 会话映射)
- **测试人员发出**："你好，我是验收测试员，我的最高级代号叫【夜枭】。"
- **预期系统反应**：机器人回复类似 "你好，夜枭测试员..."。
- **后台验证点**：
  - `SessionContextController` 成功将用户的飞书 `open_id` 映射为了系统级的逻辑 `UUID`。
  - 第一轮对话无痛持久化存入了 `RedisHotMemoryStore` 作为未来的上下文根基。

#### 🟢 回合 2：多重工具调用解析 (测试 T4 规范提取与路由)
- **测试人员发出**："我现在需要你帮我算一下 88 加上 11 等于多少？算出结果后，顺便告诉我北京今天的天气。"
- **预期系统反应**：机器人能一次性或分步在回答中给出数字 "99"，并准确汇报天气 "25度"。
- **后台验证点**：
  - 查看日志确认 `skill_hub.tool_parser` 成功利用 Pydantic V2 技术，将上文挂载的两个 Python 函数零损失地转换为 OpenAI 规范的 Function Schema。
  - MiniMax 成功发起了 Tool Call 请求并被后端路由截获执行。

#### 🟢 回合 3：短期记忆追溯限额 (测试 T4 历史记忆/LIFO策略)
- **测试人员发出**："好了，考考你的记性：我上面让你第一道计算题算的是哪两个数字？另外，你还记得我最开始告诉你的代号是什么吗？"
- **预期系统反应**：机器人精准回忆出让你算了 "88 和 11"，并且能叫出你的代号是 "夜枭"。
- **后台验证点**：
  - 触发了编排引擎的 `OE-P0-04` 和 `SM-P0-04`，由 Agent 主动从 Redis 中精准读取（拉取）了前两轮的对话记录（HotMemory) 并直接塞入了模型本次的上下文里，Tiktoken 也没有对其产生激烈的错毁性裁剪。

#### 🟢 回合 4：进程死亡与状态起死回生 (T4 全量上下文/鲁棒性验证)
- **硬核操作指引**：在不停止飞书客户端聊天界面的情况下，直接切回服务运行的终端窗口，按下 `Ctrl+C` **暴力杀死 Agent-OS 服务进程**。等待约 3 秒后，再次执行启服命令冷启动系统。
- **测试人员发出**："刚才好像机房停电了。凭你的印象，上一次我问了你哪里的天气？"
- **预期系统反应**：机器人由于记忆未丢，能够顺利回答出："上一次你问的是北京的天气。"
- **后台验证点**：
  - 在重启后本地内存全面被释放的恶劣工况下，基于 `load_runtime_state`，系统利用之前绑定的物理 `open_id` -> 找回了 `UUID` -> 继续从外部 Redis List 里无损反序列化了对话堆栈。
  - 这是 T4 阶段区别于过往单机失忆版本的核心架构跨越体验。

> 日常跑通此流程说明整个系统的 T4 “数据与上下文生命流”宣告封顶验收通过。
