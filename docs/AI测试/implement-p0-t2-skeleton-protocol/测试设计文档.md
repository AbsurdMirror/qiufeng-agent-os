# 测试设计文档：implement-p0-t2-skeleton-protocol

## 1. 文档信息
- **项目名称**：qiufeng-agent-os
- **测试规格 (Spec)**：implement-p0-t2-skeleton-protocol (P0 T2 阶段)
- **版本号**：v1.0.0
- **测试目标**：验证 T2 阶段新增的模型抽象层、热记忆存储层、编排引擎骨架（LangGraph Runtime）以及监控中心请求染色功能的正确性。

## 2. 测试范围与模块
根据 T2 规格要求与代码分析，本次测试涵盖以下模块：
1. **Model Provider (模型抽象层)**：模型提供商接口协议及 `InMemoryModelProviderClient` 的模拟调用。
2. **Storage & Memory (存储与记忆层)**：基于内存的热记忆载体实现及存取协议（滑动窗口、状态持久化）。
3. **Orchestration Engine (编排引擎层)**：`LangGraphRuntime` 的骨架加载与占位编译逻辑。
4. **Observability Hub (监控中心)**：基于 TraceID 特征的请求染色判定功能。

---

## 3. 测试项设计 (Test Cases Design)

### 3.1 模型抽象层 (Model Provider)
**测试目标**：验证模型调用契约是否标准化，以及 `InMemoryModelProviderClient` 是否能按照预期回显最后一条消息。

| 测试项 ID | 测试场景 | 前置条件 | 测试输入 | 预期输出 / 行为 |
| :--- | :--- | :--- | :--- | :--- |
| `MP-01` | 模拟客户端常规推理 | 无 | 构造 `ModelRequest`，包含多条 `ModelMessage` | 返回 `ModelResponse`，`content` 为最后一条输入消息的文本，`provider_id` 为 "in_memory"。 |
| `MP-02` | 模拟客户端空消息推理 | 无 | 构造 `ModelRequest`，`messages` 为空元组 | 返回 `ModelResponse`，`content` 为空字符串。 |

### 3.2 存储与记忆层 (Storage & Memory)
**测试目标**：验证 `InMemoryHotMemoryStore` 对底层 Redis List 语义的模拟以及高层热记忆滑动窗口管理的正确性。

| 测试项 ID | 测试场景 | 前置条件 | 测试输入 | 预期输出 / 行为 |
| :--- | :--- | :--- | :--- | :--- |
| `SM-01` | 底层原语：lpush 与 lrange | 无 | 连续 lpush 三条记录，然后 lrange(0, -1) | 数据按倒序（最新在最前）被获取，长度为 3。 |
| `SM-02` | 底层原语：ltrim 截断 | lpush 5 条数据 | 调用 `ltrim(key, 0, 2)`，再 lrange | 列表仅保留最新的 3 条数据。 |
| `SM-03` | 高层协议：append_hot_memory | 无 | 调用 append 追加记忆，设置 `max_rounds=2`，连续追加 3 条 | 返回的记忆列表长度被限制为 2，包含最新的两条记录。 |
| `SM-04` | 运行时状态持久化与加载 | 无 | 调用 `persist_runtime_state` 存入一个字典 | `load_runtime_state` 能够完整读出该字典的内容。 |

### 3.3 编排引擎层 (Orchestration Engine)
**测试目标**：验证 `LangGraphRuntime` 骨架的参数校验逻辑。

| 测试项 ID | 测试场景 | 前置条件 | 测试输入 | 预期输出 / 行为 |
| :--- | :--- | :--- | :--- | :--- |
| `OE-06` | 编译有效的入口点 | 无 | `compile_entrypoint("src.workflows:graph")` | 返回 `LangGraphExecutable`，`entrypoint` 正确，`compiled_graph` 为 None。 |
| `OE-07` | 编译无效（空）入口点 | 无 | `compile_entrypoint("   ")` | 抛出 `ValueError("invalid_langgraph_entrypoint")`。 |

### 3.4 监控中心 (Observability Hub)
**测试目标**：验证请求染色机制的判定逻辑。

| 测试项 ID | 测试场景 | 前置条件 | 测试输入 | 预期输出 / 行为 |
| :--- | :--- | :--- | :--- | :--- |
| `OB-07` | 染色请求判定（基于 context debug 字段） | 无 | `context={"is_debug": True}` | 返回 `True`。 |
| `OB-08` | 染色请求判定（基于 state 匹配） | 无 | `context={"trace_id": "t-1"}, state={"t-1"}` | 返回 `True`。 |
| `OB-09` | 普通请求判定 | 无 | `context={"trace_id": "normal"}` | 返回 `False`。 |

### 3.5 飞书实际交互手动测试扩展 (长连接模式)
**测试目标**：在 T2 阶段由于已经实现了底层的模型模拟调用（`InMemoryModelProviderClient`）和热记忆存储，通过修改本地的辅助运行脚本，可以不仅在命令行打印接受到的消息，还可以把 T2 的这几个新增模块全部串联起来，验证在收到消息时，内存记忆是否被写入，模拟模型是否被调用。

| 测试项 ID | 测试场景 | 前置条件 | 测试输入 (操作) | 预期输出 / 行为 |
| :--- | :--- | :--- | :--- | :--- |
| `M-FS-T2-01` | 发送正常消息并触发记忆与模型链路 | 本地启动长连接手动辅助脚本 | 飞书发送：“这是一条测试指令” | 控制台除了打印原始消息外，还要打印出：1. 写入热记忆的结果；2. 模拟模型调用的回显结果。 |

## 4. 测试执行计划
1. **自动化测试框架选用**：使用 Python 原生 `pytest` 框架。
2. **测试脚本目录**：在 `tests/implement-p0-t2-skeleton-protocol/tests/` 下建立自动化测试脚本。
3. **手动测试辅助脚本**：在 `tests/implement-p0-t2-skeleton-protocol/scripts/run_manual_test.py` 编写一个**特制版的启动脚本**，劫持或补充原有的 `_on_text_event` 逻辑，在其中显式调用 `model_provider` 和 `storage_memory`，扩展 T2 的观测面。
4. **输出物**：运行测试并生成本目录下的《测试结果报告_v1.0.0.md》。
