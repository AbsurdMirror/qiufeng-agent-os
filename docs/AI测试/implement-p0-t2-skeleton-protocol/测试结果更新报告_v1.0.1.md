## 2. 问题回顾与修复方案
在 `v1.0.0` 版本的手动测试中，直接在 `asyncio` 环境下调用阻塞且内部接管事件循环的飞书 SDK，导致了 `This event loop is already running` 致命错误。为了规避该问题，曾一度回退至完全同步的实现，这违背了 T2 骨架支持异步并发的架构目标。

**修复方案 (重构 `run_manual_test.py`)**：
1. **多进程隔离**：使用 `multiprocessing.Process` (spawn 模式) 将阻塞的飞书长连接客户端隔离在独立的子进程中运行。
2. **跨进程通信**：通过 `multiprocessing.Queue`，子进程在收到飞书消息后，将 `UniversalEvent` 对象传递回主进程。
3. **主进程异步消费**：主进程运行 `asyncio` 事件循环，通过 `asyncio.to_thread` 消费队列，并为每个收到的事件创建独立的 `asyncio.Task` 进行并发处理。

## 4. QA 经验与教训总结
- **架构经验**：遇到内部强制使用 `run_until_complete` 的第三方库时，必须采用多进程隔离，而不能仅仅依赖 `asyncio.to_thread`，否则会引发事件循环冲突。此经验已沉淀至 `qa-tester` Skill。
- **验证手段**：异步并发的验证不能仅停留在代码层面，必须通过“人工延迟注入 + 快速连续输入”的方式在真实（或近真实）环境中观测。
- **角色边界**：重申了 QA 测试人员不能越权修改业务源码的铁律。

> **结论**：T2 阶段的并发架构验证现已全面闭环，具备进入后续开发阶段的坚实基础。
