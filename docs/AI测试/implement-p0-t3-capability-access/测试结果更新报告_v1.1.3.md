## 3. 实际阻塞点（可复现）
### 3.1 百度安全验证（Captcha）
在部分查询组合与页面形态（尤其是移动端 m.baidu.com、或包含 `site:` 的查询）下，标题直接变为“百度安全验证”，并跳转到 `wappass.baidu.com/static/captcha/...`，此时任何点击 selector 均会 timeout。

脚本输出会出现：
- `title_before = 百度安全验证`
- `url_after = .../captcha/...`
- `error_code = browser_execution_failed`（若 click 超时）

### 3.2 结果页跳转不可稳定（无法到达 deepseek.com）
即使没有触发验证码，当前返回的页面更倾向于“deepseek 下载/入口/聚合”相关查询推荐，点击后仍停留在 baidu 搜索页（`url_after` 为 baidu 搜索 URL），不满足“进入 deepseek 官网（deepseek.com）”判定。

## 4. 证据说明（已有）
脚本在每次尝试都会输出并落盘截图，包含：
- `url_before/url_after/title_before/title_after`
- `screenshot_path`（截图路径）
- `screenshot_exists` 与 `screenshot_size_bytes`（脚本侧校验截图真实落盘）

## 5. 给开发的建议（如需让该需求稳定通过）
当前阻塞属于“真实网站反自动化策略”与“工具能力粒度”共同作用的结果。若要让该需求稳定通过，建议：
1. **支持多步骤交互**：把 `tool.browser.open` 升级为可组合动作序列（goto/fill/press/click/wait/screenshot），而非单次 goto + 单次 click。
2. **支持新标签页/弹窗**：百度结果常 `target=_blank`，需要在 Playwright 中捕获 `context.wait_for_event("page")` 并切换到新 page 继续操作，当前实现只读取同一个 `page.url`。
3. **提供反爬兼容策略（可选）**：如显式 user-agent、viewport、延时策略等；或允许 headful + Xvfb 驱动并人工通过验证码（若接受半自动）。
