# 测试设计文档：implement-p0-t1-foundation

## 1. 文档信息
- **项目名称**：qiufeng-agent-os
- **测试规格 (Spec)**：implement-p0-t1-foundation
- **版本号**：v1.0.0
- **日期**：2026-03-25
- **测试目标**：验证 P0 T1 阶段基础设施（渠道网关、编排引擎、监控中心）的核心代码逻辑是否符合规范，确保其具备正确的输入解析、接口注册和数据处理能力。

## 2. 测试范围与模块
根据代码架构检查，本次测试涵盖以下核心模块的接口与功能逻辑：
1. **Channel Gateway (渠道适配层)**：飞书 Webhook 事件接收与标准化解析 (`src/channel_gateway/feishu_webhook.py`)
2. **Orchestration Engine (编排引擎层)**：Agent 内存注册中心的注册与查询 (`src/orchestration_engine/agent_registry.py`)
3. **Observability Hub (监控中心)**：全局追踪 ID 生成与数据归一化记录 (`src/observability_hub/recording.py`)

---

## 3. 测试项设计 (Test Cases Design)

### 3.1 渠道适配层 (Channel Gateway)
**测试目标**：验证飞书 Webhook 负载能否正确响应握手挑战，以及 `TextEventParserFactory` 对不同模式（Webhook / Long Connection）事件的正确路由和提取。

| 测试项 ID | 测试场景 | 前置条件 | 测试输入 (Payload) | 预期输出 / 行为 |
| :--- | :--- | :--- | :--- | :--- |
| `CG-01` | 处理飞书 URL 验证挑战 | 无 | 包含 `type="url_verification"` 和 `challenge="xxx"` 的字典 | 返回 `FeishuWebhookResult`，`is_challenge=True`，且 `challenge="xxx"`，`event=None`。 |
| `CG-02` | Webhook模式解析合法文本消息 | 无 | 完整且合法的飞书文本消息 JSON | 通过工厂获取 `webhook` 解析器并成功提取 `UniversalTextEvent` 对象。 |
| `CG-03` | 长连接模式解析合法文本消息 | 无 | 完整且合法的飞书文本消息 JSON | 通过工厂获取 `long_connection` 解析器并成功提取 `UniversalTextEvent` 对象。 |
| `CG-04` | 工厂模式获取不支持的解析器 | 无 | `TextEventParserFactory.get("feishu", "unknown")` | 抛出 `ValueError("unsupported_event_parser")`。 |
| `CG-05` | Webhook异常处理：非文本类型 | 无 | 飞书消息中 `message_type="image"` | 抛出 `ValueError("unsupported_message_type")`。 |
| `CG-06` | Webhook异常处理：不支持的事件 | 无 | 飞书消息中 `event_type="unknown.event"` | 抛出 `ValueError("unsupported_event_type")`。 |

### 3.2 编排引擎层 (Orchestration Engine)
**测试目标**：验证 `InMemoryAgentRegistry` 能够正常实现 `AgentSpec` 协议的注册校验以及灵活的版本查询。

| 测试项 ID | 测试场景 | 前置条件 | 测试输入 | 预期输出 / 行为 |
| :--- | :--- | :--- | :--- | :--- |
| `OE-01` | 成功注册合法的 Agent | 注册中心为空 | 构造包含必填字段的 `AgentSpec` 对象 | 注册成功，可通过 `get` 或 `list_all` 查看到该 Agent。 |
| `OE-02` | 注册校验：字段为空 | 无 | `AgentSpec` 中的 `agent_id` 或 `version` 等核心字段为空字符串 | 抛出 `ValueError` (如 `invalid_agent_id`)。 |
| `OE-03` | 指定版本号获取 Agent | 已注册同一 Agent 的 "v1.0" 和 "v2.0" | `get(agent_id, tenant_id, version="v1.0")` | 返回对应 "v1.0" 版本的 `AgentSpec` 对象。 |
| `OE-04` | 不指定版本号获取最新 Agent | 已注册同一 Agent 的 "v1.0" 和 "v2.0" | `get(agent_id, tenant_id, version=None)` | 根据降序排列，自动返回 "v2.0" 版本的 `AgentSpec` 对象。 |
| `OE-05` | 按租户列表查询 | 注册不同 tenant_id 的多个 Agent | `list_by_tenant(tenant_id="tenant_A")` | 仅返回属于 "tenant_A" 的 Agent 列表。 |

### 3.3 监控中心 (Observability Hub)
**测试目标**：验证 `GlobalTraceIDGenerator` 的高并发唯一性和 `record()` 函数对于异构数据的打平 (Flatten) 归一化处理能力。

| 测试项 ID | 测试场景 | 前置条件 | 测试输入 | 预期输出 / 行为 |
| :--- | :--- | :--- | :--- | :--- |
| `OB-01` | 生成追踪 ID 的基本格式 | 无 | 调用 `generate_trace_id()` | 返回符合 `prefix-timestamp-sequence-uuid` 格式的字符串。 |
| `OB-02` | 同毫秒内生成多个追踪 ID | 模拟时间戳不变 | 连续调用 `generate()` 两次 | 第二次的 `sequence` 必须比第一次大 1。 |
| `OB-03` | 记录归一化：处理字符串数据 | 无 | `record(trace_id="123", data="error msg")` | 生成的记录 payload 为 `{"message": "error msg"}`，payload_type="str"。 |
| `OB-04` | 记录归一化：处理深层嵌套字典 | 无 | `data={"user": {"id": 1, "info": {"name": "test"}}}` | payload 展开为一维字典：`{"user.id": 1, "user.info.name": "test"}`，payload_type="dict"。 |
| `OB-05` | 记录归一化：处理 Pydantic/BaseModel | 无 | 传入实现了 `dict()` 或 `model_dump()` 的类实例 | 实例被序列化后打平，payload_type="basemodel"。 |
| `OB-06` | 异常处理：不支持的数据类型 | 无 | 传入整数或列表 | 抛出 `TypeError("unsupported_record_data_type")`。 |

### 3.4 飞书实际交互手动测试 (长连接模式)
**测试目标**：在本地启动飞书长连接模式下，通过真实飞书客户端发送不同类型消息，验证系统的接收、解析和异常兼容能力。

| 测试项 ID | 测试场景 | 前置条件 | 测试输入 (操作) | 预期输出 / 行为 |
| :--- | :--- | :--- | :--- | :--- |
| `M-FS-01` | 发送纯文本消息 | 本地启动长连接，机器人在线 | 在飞书客户端向机器人发送纯文本（如“你好”） | 系统控制台成功打印接收事件，正常解析出 `message_type="text"` 和文本内容。 |
| `M-FS-02` | 发送富文本消息 (含链接/代码) | 同上 | 发送包含超链接、@提及或代码块的富文本消息 | 系统正常接收并解析，能提取文本内容或进行安全降级处理，不发生崩溃。 |
| `M-FS-03` | 发送图片消息 | 同上 | 在飞书客户端向机器人发送一张图片 | 系统识别为非纯文本，抛出或记录不支持的消息类型警告（如 `unsupported_message_type`），维持长连接不断开。 |
| `M-FS-04` | 发送语音消息 | 同上 | 在飞书客户端向机器人发送一段语音 | 系统识别类型为语音，记录不支持的类型警告，程序不崩溃，继续等待下一条消息。 |

## 4. 测试执行计划
1. **自动化测试框架选用**：使用 Python 原生 `pytest` 框架编写单元测试脚本。
2. **测试脚本目录**：放置于代码库的 `tests/` 目录下。
3. **手动联调环境准备**：配置本地飞书 `APP_ID` 和 `APP_SECRET`，运行长连接启动入口。
4. **输出物**：执行自动化代码测试与手动飞书长连接交互测试后，生成本目录下的《测试结果报告_v1.0.0.md》。
