## 3. 手动测试结果 (Manual Tests - 飞书长连接)
基于 `scripts/run_manual_test.py` 启动本地长连接环境，结合真实的飞书客户端进行了多场景消息发送测试。

| 测试项 ID | 测试场景与输入 | 实际输出与表现 | 执行结果 |
| :--- | :--- | :--- | :--- |
| `M-FS-01` | **发送纯文本**<br>输入: `"你好12345"` | 控制台正常打印解析后的 JSON，提取出 `"text": "你好12345"`。 | ✅ Pass |
| `M-FS-02a` | **发送带表情包的纯文本**<br>输入: `"嘿嘿嘿 [笑哭][了解][看][看]"` | 控制台正常解析出带有表情占位符的字符串，格式正确。 | ✅ Pass |
| `M-FS-02b` | **发送富文本格式的纯文本**<br>输入: 粗体和斜体格式文本 | 飞书开放平台底层 SDK 抛出 `unsupported_message_type` 错误，但长连接保持健壮，**未发生程序崩溃**，继续监听。 | ✅ Pass |
| `M-FS-03` | **发送图片**<br>操作: 发送一张图片文件 | 飞书底层 SDK 识别为非文本类型，触发 `unsupported_message_type` 错误，长连接捕获异常后**未崩溃断开**。 | ✅ Pass |

> **发现与建议**：
> 针对富文本消息（如粗体/斜体格式文本），飞书底层协议并未将其作为简单的 `message_type="text"` 处理（可能为 `post` 等类型）。在当前的 T1 阶段实现中，此类消息会被安全地拦截（`unsupported_message_type`），这符合当前只支持纯文本（Plain Text）的设计预期，同时证明了异常捕获机制可以有效保障长连接服务的高可用性（不崩溃）。
