## ---

**第二章：T1 阶段 —— 基础设施与通道建立**

本阶段的目标是打通系统的“入水口”和监控底座，确保请求进得来且可追踪。

### **📋 Todo List**

- \[ ] **GW-P0-01** 框架集成：集成 NoneBot2 基础框架。
- \[ ] **GW-P0-02** 链路调通：调通飞书长连接（WebSocket）并实现文本提问接收。
- \[ ] **OE-P0-01** 注册接口：定义支持元数据与身份声明的 Agent 注册接口。
- \[ ] **OB-P0-01** TraceID 生成：实现全局唯一的 trace\_id 生成机制。
- \[ ] **OB-P0-03** 归一化采集：提供统一的 record() 接口，屏蔽底层存储差异。

### **🔗 Spec 关联描述**

T1 阶段是系统的**物理启点**。**GW**（渠道层）负责通过飞书长连接通道接收原始流量，这是所有后续动作的诱因。在请求进入的第一时间，**OB**（监控层）必须介入并生成 trace\_id，这个 ID 将作为唯一“染色标签”被注入到 **OE**（编排层）的注册接口中。此时的编排层虽然还没有逻辑，但它必须能识别这个 ID，从而在监控日志中标记出“一个新任务已经由飞书开启”。

## ---

**第三章：T2 阶段 —— 编排骨架与状态协议**

本阶段构建系统的“脊椎”，确立数据如何转换以及状态如何暂存。

### **📋 Todo List**

- \[ ] **GW-P0-03** 结构定义：定义系统通用 UniversalEvent 数据结构。
- \[ ] **GW-P0-04** 消息转换：实现飞书原始消息向通用结构的映射转化。
- \[ ] **OE-P0-02** 基类定义：定义 BaseOrchestrator 抽象基类以规范执行接口。
- \[ ] **OE-P0-03** 引擎集成：集成 LangGraph 核心并支持有状态图的构建编译。
- \[ ] **MP-P0-08** 接口适配：实现并适配与编排引擎层之间的同步交互接口。
- \[ ] **SM-P0-01** 热记忆载体：集成 Redis 实现热记忆（Hot Memory）的物理存储。
- \[ ] **SM-P0-05** 存取协议：定义并实现标准的数据存取协议。
- \[ ] **OB-P0-02** 请求染色：实现染色标记判定逻辑。

### **🔗 Spec 关联描述**

这是**协议对齐**的关键期。**GW** 将飞书消息转换为标准的 UniversalEvent 后，送入 **OE**。**OE** 此时基于 **LangGraph** 建立起逻辑图的骨架，并通过 **MP** 的接口适配向模型发起第一次握手。为了保证编排过程不丢失，**OE** 产生的每一个节点状态都需要通过 **SM**（存储层）的存取协议实时同步到 **Redis**（热记忆载体）中。同时，**OB** 根据 isDebug 标记决定是否对此流程进行全量染色记录。

## ---

**第四章：T3 阶段 —— 外部能力接入与转发**

本阶段接通系统的“手脚”，让 Agent 具备操作浏览器和调用特定模型的能力。

### **📋 Todo List**

- \[ ] **OE-P0-07** 契约定义：定义 Capability 接口的获取、调用与返回协议。
- \[ ] **SH-P0-03** 浏览器工具开发：基于 browser-use 开发浏览器原子工具。
- \[ ] **SH-P0-04** 转发路由实现：基于 Capability 接口实现编排层到 PyTools 的转发。
- \[ ] **MP-P0-03** 格式归一化：集成 LiteLLM 实现不同 API 协议向统一 Message 格式的转换。
- \[ ] **MP-P0-04** MiniMax 适配：完成 MiniMax 模型的驱动适配。

### **🔗 Spec 关联描述**

T3 阶段是系统的**执行力闭环**。核心在于 **OE-P0-07** 定义的 Capability 契约。**SH**（工具层）和 **MP**（模型层）必须严格遵守该契约：**SH** 封装好 browser-use 后，通过转发路由挂载到契约接口上；**MP** 则通过 **LiteLLM** 将 MiniMax 的私有协议翻译成契约要求的统一格式。这一阶段完成后，Agent 已经可以执行“去网页搜索并返回原始数据”的动作。

## ---

**第五章：T4 阶段 —— 数据流转与上下文管理**

本阶段赋予系统“短期记忆”和“上下文感知”能力，处理复杂的业务数据流。

### **📋 Todo List**

- \[ ] **GW-P0-05** 身份映射：实现 open\_id 到系统逻辑 UUID 的转换。
- \[ ] **GW-P0-06** 消息去重：实现基于指纹的消息去重与幂等机制。
- \[ ] **OE-P0-04** 记忆读取：实现历史记忆的拉取、解析与状态转换。
- \[ ] **OE-P0-05** 上下文维护：实现运行时 RuntimeContext 的动态更新与同步。
- \[ ] **OE-P0-06** 状态持久化：实现执行上下文向存储与记忆层的异步持久化。
- \[ ] **SH-P0-02** 工具规范解析：基于 Doxygen 自动生成符合 LLM 规范的 Tool Spec。
- \[ ] **MP-P0-01** 名称匹配：实现基于物理名称的直接调度匹配机制。
- \[ ] **MP-P0-02** 自动裁剪：集成 Tiktoken 实现基于窗口的物理裁剪。
- \[ ] **SM-P0-02** 热记忆策略：实现基于 LIFO 的最近 N 轮对话缓存。
- \[ ] **SM-P0-03** 持久化存储：实现对话报文与调研报告的无损持久化存入。
- \[ ] **SM-P0-04** 上下文注入：提供被动注入接口供编排引擎初始化节点状态。

### **🔗 Spec 关联描述**

这是**复杂度最高**的阶段。**OE** 在运行时，需要通过 **SM** 的上下文注入拉取历史数据（记忆读取），并根据 **MP** 的自动裁剪功能确保这些数据不会撑爆 Token 限制。**SH** 此时必须交付工具描述（Tool Spec），让 **MP** 知道当前有哪些浏览器工具可用。所有这些上下文变更最终由 **OE-P0-06** 汇总并触发 **SM** 的持久化存储。**GW** 层则通过身份映射，确保这些记忆能精准地匹配到飞书上的特定用户。

## ---

**第六章：T5 阶段 —— 系统健壮、安全与调试**

本阶段为系统穿上“盔甲”，确保输出质量并提供开发者调试工具。

### **📋 Todo List**

- \[ ] **GW-P0-07** 响应原语：定义 ReplyText 纯文本响应标准。
- \[ ] **GW-P0-08** 异步接口：实现可随时调用的异步消息回传接口。
- \[ ] **GW-P0-09** 渠道适配：实现飞书渠道的消息投递。
- \[ ] **SH-P0-01** 安全原语实现：限制工具的资源访问与库调用权限。
- \[ ] **MP-P0-05** 定义结构：规范化模型输出结构与系统提示词模板。
- \[ ] **MP-P0-06** 强校验：集成 Pydantic 进行类型与格式验证。
- \[ ] **MP-P0-07** 基础自愈：在校验失败时尝试自动重试。
- \[ ] **OB-P0-04** 调试引擎实现：集成 JSONL 格式存储大载荷调试数据。
- \[ ] **OB-P0-05** 滚动清理：实现基于容量或时间的日志清理。
- \[ ] **OB-P0-06** CLI 实时日志：提供基于 TraceID 过滤的实时调试输出。

### **🔗 Spec 关联描述**

T5 阶段是系统的**交付保障**。**MP** 通过 **Pydantic 强校验**和**自愈机制**确保调研报告的 JSON 格式不破碎。**SH** 启用**安全原语**防止浏览器工具被恶意利用。**GW** 实现**异步接口**，让 Agent 在浏览器抓取的漫长过程中能不断向飞书回传进度。最后，**OB** 的 **JSONL 调试引擎**和 **CLI 日志**为开发者提供了观察这个复杂黑匣子的终极手段。
