### **技能与工具层 (Skill Hub) 架构设计规格书**

#### **第一章：模块概述与子模块组成**

技能与工具层（Skill Hub）是 Agent-OS 的核心执行端，负责为模型提供认知指引与物理操作能力。本层通过解耦专家指令集（Skills）与原子执行工具（MCP/PyTools），构建了一套兼顾生态兼容性与极致安全性的双轨执行体系。

**1.1 设计目标**

* **双轨并行执行**：支持基于 MCP 协议的外部工具生态与基于 Python 的内部受控工具集并行运作。  
* **物理安全隔离**：通过常驻 Docker 容器划定外部工具的执行边界，保护宿主机内核安全。  
* **逻辑审计管控**：对内部工具实施 AST 静态检查与运行时环境阉割，实现代码级安全。  
* **共享空间协同**：建立基于宿主机挂载的统一工作区，支持不同类型的工具在同一会话空间内接力协作。

**1.2 子模块构成**

* **Skills 模块 (Skills Module)**：存放专家 SOP、指令集及行为规范的静态资源库。  
* **MCP 模块 (MCP Module)**：负责管理与运行非受信任第三方工具的常驻 Docker 沙盒环境。  
* **PyTools 模块 (PyTools Module)**：负责运行内部研发或经过严格审计的受监控 Python 高级工具集。  
* **系统交互模块 (System Interaction Module)**：负责能力发现、工具 Schema 注入、请求路由分发及会话生命周期管理。

#### ---

**第二章：Skills 模块 (Skills Module)**

本模块作为 Agent 的“知识锦囊”，负责存储和提供改变模型推理偏好与操作逻辑的提示词补丁。

**2.1 存储与管理规范**

* **资源结构**：采用 SKILL.md 标准格式，以目录形式组织。每个技能包必须包含背景（Background）、标准作业程序（Instructions）、执行禁忌（Constraints）及少样本示例（Examples）。  
* **加载逻辑**：模块在初始化时扫描技能目录，提取各技能的元数据并建立索引。

**2.2 交互逻辑**

* **按需读取**：该模块仅提供 get\_skill(name) 接口。模型在发现任务超出基础能力范围时，主动调用该接口获取完整的 Markdown 格式操作指南。  
* **非执行特性**：本模块不涉及任何代码运行逻辑，其产生的输出仅作为高阶系统提示词注入模型上下文，引导模型后续调用 MCP 或 PyTools。

#### ---

**第三章：MCP 模块 (MCP Module)**

本模块作为系统的“外部能力沙盒”，通过标准 MCP 协议接入并运行庞大的第三方工具生态。

**3.1 运行环境与空间管理**

* **常驻 Docker 沙盒**：所有 MCP Server 统一部署并运行在一个常驻的 Docker 容器中。该容器预置 Node.js、Python 等主流运行时，与宿主机环境物理隔离。  
* **共享工作区挂载**：容器启动时挂载宿主机的会话工作区目录（/data/agent\_workspace/{session\_id}）至容器内的 /workspace，作为 MCP 工具唯一的物理操作对象。

**3.2 安全与审计机制**

* **指令拦截与过滤**：模块内置基于正则表达式的命令拦截器，对所有通过 shell 接口传入的字符串进行预检，封禁涉及宿主机逃逸、系统配置修改等高危动作。  
* **审计追踪**：实时记录容器内所有进程的启动参数、环境变量快照及标准输出流，为安全复盘提供全量日志。  
* **风险告知**：在调用非受信任 MCP 资源前，由模块触发安全声明，确保用户在知情前提下授予执行权。

#### ---

**第四章：PyTools 模块 (PyTools Module)**

本模块作为系统的“受信任执行路径”，负责运行经过内部研发或严格过审的高安全级别工具。

**4.1 准入与开发规范**

* **Doxygen 定义注入**：开发者使用标准 Python 函数编写工具，利用 Doxygen 格式的函数头注释定义工具名称、描述及参数约束，模块自动解析该信息并转化为 LLM 的 Tool Spec。  
* **三级审计流水线**：  
  * **第一级 (AST 静态审核)**：严禁源码中出现 os、subprocess、socket 等原生危险库的导入，严禁 exec、eval 等动态执行语句。  
  * **第二级 (人工复核)**：后期开放的第三方 PyTools 必须经过人工逻辑审查，标记为 Verified 后方可进入生产环境。  
  * **第三级 (环境阉割)**：工具运行在受限的 Python 解释器中，禁用 \_\_import\_\_ 等动态加载能力。

**4.2 受控 SDK 与监听机制**

* **安全原语接口**：PyTools 必须调用系统提供的受控 SDK（如 secure\_fs、secure\_shell）进行底层操作。  
* **深度监听**：SDK 内部集成了实时监控钩子，对每一次文件读写、每一条命令执行进行逻辑校验（如检查路径是否超出 /workspace）和行为审计。  
* **共享空间协同**：PyTools 直接在宿主机对应的会话工作区目录执行操作，实现与 MCP 模块产生的数据无缝对接。

#### ---

**第五章：系统交互模块 (System Interaction Module)**

本模块作为第四层的对外网关与控制中心，负责协调模型、技能库与两个执行路径之间的交互逻辑。

**5.1 能力路由与注入**

* **统一清单暴露**：扫描 Skills 库、MCP 容器与 PyTools 库，将静态指令与原子工具合并，向模型层返回统一的 JSON Schema 菜单。  
* **冲突回避策略**：若 PyTools 库与 MCP 库存在功能重叠，模块优先隐藏 MCP 工具，向模型推送受监控的 PyTools 路径。  
* **请求分发**：根据模型下达的 tool\_use 指令前缀，准确将请求分发至对应的执行子模块。

**5.2 安全确认与生命周期管理**

* **凭证化执行 (Action Ticket)**：当 MCP 或 PyTools 触发安全策略拦截时，交互模块捕获异常并生成唯一的 Action\_Ticket。模型接收错误后，需引导用户完成授权签字。用户授权后，模型携带 Approved Ticket 再次请求，交互模块校验凭证有效性后放行。  
* **会话空间清理**：模块实时监听编排层的会话结束信号。当会话终止时，自动触发 /workspace/{session\_id} 目录的物理擦除，确保数据不发生跨会话污染。
