# **存储与记忆层 (Storage & Memory) 架构设计规格书**

## **第一章：模块概述与子模块组成**

存储与记忆层是 Agent-OS 的“长期记忆体”与“认知中心”。它不仅负责原始对话数据的无损持久化，还通过分级存储（热、温、冷）策略与动态特征提取技术，为编排引擎提供多粒度的上下文支持。本层通过将“提问”视为核心特征源，实现了对用户意图的深度理解与长期跟踪。

### **1.1 设计目标**

* **认知深度**：支持跨度长达数月的历史回溯与用户画像演进。  
* **成本优化**：通过文本压缩与异步摘要技术，大幅降低云端 API 的 Token 消耗与存储成本。  
* **检索精准度**：结合向量搜索与结构化 KV 过滤，提供高信噪比的记忆召回。  
* **主动性**：支持 Agent 通过工具层主动调取底层无损记录，进行事实核查。

### **1.2 子模块构成**

本层由以下四个功能单元组成：

1. **全量历史数据库 (Full History DB)**：负责所有对话的无损记录与高压缩比持久化。  
2. **分级上下文管理器 (Context Tiering Manager)**：管理热记忆（实时）、温记忆（当日）与冷记忆（历史简报）的流转与生命周期。  
3. **用户画像模块 (User Profiling Module)**：从用户提问中提取结构化特征（KV），并建立意图向量索引。  
4. **系统交互网关 (System Interaction Gateway)**：提供被动上下文注入接口，并将记忆查询能力封装为工具暴露给 Skill Hub。

## ---

**第二章：全量历史数据库 (Full History DB)**

本模块是系统的“单一事实来源（Source of Truth）”，存储所有原始交互报文。

### **2.1 存储结构**

采用关系型数据库（如 PostgreSQL）存储，以 session\_id 作为核心聚集索引。

* **表字段定义**：  
  * id: 唯一消息 ID (UUID)。  
  * time: 毫秒级时间戳。  
  * tokencount: 该条记录消耗的 Token 数（由模型层回传）。  
  * Q: 用户提问文本（压缩存储）。  
  * A: 模型回答文本（压缩存储）。

### **2.2 存储优化**

* **无损压缩协议**：在应用层写入前使用 Zstandard (zstd) 或 LZ4 算法对 Q 和 A 字段进行压缩。读取时自动解压。  
* **冷热分离**：超过 90 天的原始记录可迁移至冷存储（如 OSS），数据库内仅保留元数据。

## ---

**第三章：分级上下文管理器 (Context Tiering Manager)**

本模块负责根据时间时效性和访问频率，将数据划分为三个层级。

### **3.1 热记忆 (Hot Memory)**

* **载体**：Redis List。  
* **策略**：保留最近 5-10 轮对话原文。  
* **限制**：受 Token 上限与轮数上限双重控制（以先到者为准）。

### **3.2 温记忆 (Warm Memory)**

* **载体**：向量数据库（当日分区）。  
* **内容**：今日内发生的所有全量对话片段。  
* **特性**：不进行语义压缩，通过向量检索支持对“刚刚聊过”的内容进行快速定位。

### **3.3 冷记忆 (Cold Memory)**

* **载体**：向量数据库（长期索引）+ 关系型数据库。  
* **压缩流程 (Daily Digest)**：  
  1. **触发**：每日凌晨异步启动。  
  2. **提取**：从全量库拉取昨日该 Session 的所有 QA。  
  3. **压缩**：调用 LLM 生成“日记式”语义摘要。  
  4. **持久化**：将摘要向量化存入冷记忆库，并关联日期标签。

## ---

**第四章：用户画像模块 (User Profiling Module)**

本模块通过分析用户的提问行为，构建动态的、可进化的用户特征模型。

### **4.1 提问特征提取 (Feature Extraction)**

* **KV 存储机制**：记录用户提问表现出的特征，如 {"coding语言": "python"}。  
* **增量列表**：当日内所有的特征以列表形式追加：\[{"语言": "py"}, {"语言": "go"}\]。  
* **合并算法**：每日定时触发 LLM 任务，将重复或冲突的特征进行语义合并。例如将“初学者”与“进阶”合并为最新的“进阶状态”。

### **4.2 意图向量库 (Intent Index)**

* **内容**：将用户输入的所有**提问原文**进行全量向量化。  
* **用途**：当用户说“我以前问过...”时，通过语义匹配直接找到历史对话 ID。

## ---

**第五章：系统交互网关 (System Interaction Gateway)**

本模块定义了编排层如何消费记忆层能力的两种路径。

### **5.1 被动上下文注入 (Passive Context Injection)**

在编排层发起推理请求前，网关自动组装以下 Prompt 补丁：

* **Profile Patch**：当前用户最新的 KV 特征描述。  
* **Memory Snippets**：基于当前输入，从冷/温记忆库召回的相似片段。  
* **Hot Flow**：Redis 中的最近对话流。

### **5.2 主动查询工具 SDK (Active Memory SDK)**

将存储层的能力封装为标准工具（PyTools），注册到 **Skill Hub**。编排层可根据推理需要主动调用：

* get\_raw\_history(dialogue\_id)：从全量库提取并解压缩原始 QA 对。  
* search\_cold\_memory(query)：跨越时间长河搜索历史摘要。  
* search\_warm\_memory(query)：检索今日细节。  
* search\_past\_intents(query)：匹配用户曾表达过的类似意图及其对应的对话 ID。

### **5.3 交互准则**

* **层间互联原则**：被动注入走“编排层-存储层”直连通道（追求性能）；主动查询走“编排层-工具层-存储层”链路（追求审计与逻辑统一）。
