# **全栈监控与治理中心 (Observability Hub) 架构设计规格书**

## **第一章：模块概述与子模块组成**

全栈监控与治理中心是 Agent-OS 的“神经中枢”与“黑匣子”。它作为底座模块，穿插在系统所有层级中，负责记录、分析并展示 Agent 从接收请求到输出结果的全生命周期轨迹。

### **1.1 设计目标**

* **极致透明度**：支持对 Agent 推理链条（CoT）、工具执行和存储存取的全链路追踪（Tracing）。  
* **成本可控性**：精准记录 Token 消耗、API 费用及系统资源占用，支持按维度统计。  
* **动态调试性**：引入“请求染色”机制，支持在生产环境下按需开启全量详细日志，而非常态化存储冗余数据。  
* **低侵入性**：通过异步上报和非阻塞 I/O，确保监控逻辑不拖慢 Agent 的核心推理速度。

### **1.2 子模块构成**

本层由以下三个核心单元组成：

1. **混合存储模块 (Hybrid Storage Module)**：采用“时序数据库 \+ 本地日志文件”的双轨制存储，平衡统计需求与调试需求。  
2. **系统交互引擎 (System Interaction Engine)**：负责维护“染色表”，提供归一化的上报接口，并根据染色状态执行数据路由。  
3. **开发交互中心 (Developer Interaction Center)**：初期提供基于 CLI 的结构化实时日志查看，后期扩展为基于 Grafana 的可视化看板。

## ---

**第二章：混合存储模块 (Hybrid Storage Module)**

为了解决长文本（Raw Prompt）占用空间大与监控指标需要高频聚合查询的矛盾，本层放弃了单一数据库方案，转而采用混合存储。

### **2.1 统计引擎：InfluxDB (TSDB)**

* **存储对象**：高频、短小的结构化指标（Metrics）。  
* **核心字段**：trace\_id, agent\_id, input\_tokens, output\_tokens, duration\_ms, status\_code, tool\_name。  
* **优势**：支持大规模并发写入，支持高效的范围查询（如：计算过去 24 小时某 Agent 的总成本）。  
* **策略**：设置较长的 TTL（如 30-90 天），用于趋势分析。

### **2.2 调试引擎：Local Log Files (JSONL)**

* **存储对象**：长文本、大载荷的调试详情（Debug Traces）。  
* **核心内容**：trace\_id 关联下的全量 Raw Prompt, Model Response, Tool Observation, Memory Context。  
* **存储格式**：JSON Lines。每行一条记录，方便利用 grep, awk, jq 等工具秒级检索。  
* **策略**：仅在“染色模式”开启时触发写入。设置滚动清理策略（如保留 24 小时或固定 10GB 空间），确保不撑爆磁盘。

**第三章：系统交互引擎 (System Interaction Engine)**

本模块是监控层与其他层级通信的唯一入口，其核心逻辑在于根据“染色标记”决定数据的去向。

### **3.1 请求染色机制 (Request Coloring)**

* **染色表定义**：记录正在被监控的 trace\_id 或 session\_id。  
* **触发逻辑**：  
  * **主动染色**：开发者在 API Header 或对话指令中加入 isDebug=True。  
  * **自动染色**：当模型报错或发生 Failover 时，系统自动将当前 trace\_id 加入染色记录。  
* **传递机制**：染色标记随 RuntimeContext 在各层间传递，确保整条链路对“是否开启全量记录”达成共识。

### **3.2 归一化采集接口**

提供屏蔽底层差异的通用接口，支持多种数据格式输入：

* **标准接口**：record(trace\_id, data, level: LogLevel)  
* **数据兼容性**：  
  * data 为 dict：自动扁平化处理，提取指标至 InfluxDB。  
  * data 为 str：若处于染色模式，直接追加至本地日志。  
  * data 为 BaseModel：利用 Pydantic 自动序列化。

## ---

**第四章：开发交互中心 (Developer Interaction Center)**

本模块负责将冷冰冰的数据转化为开发者可读的洞察。

### **4.1 阶段一：极客调试工具 (CLI \+ Log)**

* **结构化输出**：所有本地日志必须包含统一的 JSON 头。  
* **实时观测**：开发者可以使用以下命令进行现场复盘：  
  tail \-f agent\_debug.log | jq 'select(.trace\_id \== "target\_id")'  
* **优势**：零部署成本，极致响应速度。

### **4.2 阶段二：可视化看板 (Visualization)**

* **Grafana 集成**：通过配置 InfluxDB 数据源，构建多维度的 Agent 运行状态大屏。  
* **核心面板设计**：  
  * **Token 实时水位线**：各模型供应商的余额与消耗趋势。  
  * **延迟分布火焰图**：定位模型、工具、存储哪个环节最耗时。  
  * **自愈成功率**：统计模型在输出破碎后通过自愈机制恢复的比例。

## ---

**第五章：执行流与数据走向逻辑**

1. **产生请求**：渠道适配层生成 trace\_id。  
2. **状态判定**：系统交互引擎检查此 ID 是否染色。  
3. **分发逻辑**：  
   * **所有模式下**：层 3/4/5 的性能指标 \-\> 异步队列 \-\> **InfluxDB**。  
   * **仅染色模式下**：层 3/4/5 的原始输入输出 \-\> 异步序列化 \-\> **Local Log File**。  
4. **消费阶段**：开发者通过 Grafana 发现异常趋势，通过 trace\_id 回溯 Local Log 获取故障现场。
