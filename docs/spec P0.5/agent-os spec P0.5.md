### **第一章：P0.5 阶段总体规格概览**

本章节从系统全局视角描述 `P0.5` 阶段的目标边界。

`P0.5` 的定位不是新增复杂功能，也不是对 `P0` 六层能力进行实质重写，而是在 **保持现有能力语义与行为基本不变** 的前提下，完成：

1. `app` 层的对外封装与统一暴露；
2. 六层模块内部的目录结构整理与职责显性化；
3. 为后续 `P1` 阶段的能力升级提前清理组织结构与装配边界。

| specId | 层级名称 | P0.5 总体规格描述 |
| :---- | :---- | :---- |
| SYS-P05-01 | **应用封装层 (App Facade)** | 在不改变 `P0` 现有运行链路语义的前提下，将 `app` 重构为六层模块的统一门面层，对外提供清晰的封装入口与装配出口。 |
| SYS-P05-02 | **渠道适配层 (Channel Gateway)** | 不新增渠道能力，不修改既有飞书链路语义，仅对文件组织、解析器/发送器/会话模块进行目录整理与职责显性化。 |
| SYS-P05-03 | **编排引擎层 (Orchestration Engine)** | 不实质改造编排能力，仅整理编排基类、运行时、上下文管理与注册中心等文件组织方式，强化模块可读性与后续演进承载力。 |
| SYS-P05-04 | **技能与工具层 (Skill Hub)** | 在不改变现有 PyTools 与能力路由行为的前提下，重组目录结构，显式区分框架核心、内置安全原语与内置工具实现。 |
| SYS-P05-05 | **模型抽象层 (Model Provider)** | 不新增模型能力与路由策略，仅整理 Provider 适配器、路由器、校验器等模块布局，为后续多模型扩展预留清晰结构。 |
| SYS-P05-06 | **存储与记忆层 (Storage & Memory)** | 不新增存储能力，仅对协议、后端实现、导出层的目录布局进行必要整理，保持现有热记忆与状态接口语义稳定。 |
| SYS-P05-07 | **全栈监控与治理中心 (Observability Hub)** | 不新增监控能力，仅整理记录、染色、JSONL 与 CLI 工具的模块边界，保持当前观测接口与数据语义不变。 |

**第二章：应用封装层 (App Facade) P0.5 规格详述**

| specId | 对应子模块 | spec 描述 |
| :---- | :---- | :---- |
| APP-P05-01 | 应用门面 (Facade) | **统一入口**：将当前分散的 `app` 启动、配置、运行器、CLI 逻辑重新组织为统一门面层，对外提供清晰、稳定的应用入口。 |
| APP-P05-02 | 六层装配封装 (Module Assembly) | **装配收口**：保留当前六层 `initialize()` 与 `build_application()` 语义不变，但在 `app` 层形成更清晰的装配组织结构。 |
| APP-P05-03 | 运行器拆分 (Runners) | **运行器整理**：将长连接运行、Webhook 运行等入口函数拆分到更明确的子目录，减少 `app` 顶层文件混杂度。 |
| APP-P05-04 | CLI 入口整理 (CLI Entry) | **入口清晰化**：将命令行解析、配置命令与启动命令组织为更清晰的 CLI 入口结构，保持现有命令语义与行为不变。 |
| APP-P05-05 | 配置接口收束 (Config Facade) | **配置收束**：保留当前配置加载与持久化逻辑，但在 `app` 层形成统一配置访问出口，避免配置能力散落。 |
| APP-P05-06 | 对外暴露规范 (Public Exports) | **暴露规范**：明确 `app` 层哪些对象面向框架使用者暴露，哪些仅为内部实现，降低后续 `P1` 公共 API 设计成本。 |
| APP-P05-07 | 兼容性约束 (Compatibility) | **兼容优先**：在完成目录与封装重组时，尽量保持现有运行命令、启动方式与核心导入链不被破坏。 |

**第三章：渠道适配层 (Channel Gateway) P0.5 规格详述**

| specId | 对应子模块 | spec 描述 |
| :---- | :---- | :---- |
| GW-P05-01 | 目录重组 (Directory Layout) | **结构整理**：将事件模型、解析器、发送器、传输实现、会话上下文等按职责重组为更清晰的目录结构。 |
| GW-P05-02 | 领域对象保持稳定 (Stable Domain Objects) | **语义保持**：保持 `UniversalEvent`、`ReplyText` 等现有领域对象定义与行为语义稳定，不在 `P0.5` 阶段新增响应类型或修改字段语义。 |
| GW-P05-03 | 解析器组织优化 (Parser Organization) | **解析器显性化**：将 Webhook / 长连接解析逻辑的组织方式显性化，便于未来新增渠道或传输方式时复用现有结构。 |
| GW-P05-04 | 发送器组织优化 (Sender Organization) | **发送器显性化**：保留现有飞书发送能力不变，仅将发送器实现与响应原语的边界组织得更清晰。 |
| GW-P05-05 | 会话模块独立 (Session Module) | **会话独立**：将身份映射、消息去重、会话上下文控制逻辑以更明确的模块形式组织，避免职责继续散落。 |
| GW-P05-06 | 入口与实现分离 (Entry vs Impl) | **入口分离**：将供 `app` 层调用的稳定入口与具体渠道实现细节分离，减少上层对底层文件位置的直接依赖。 |
| GW-P05-07 | 无功能增量约束 (No Feature Expansion) | **边界约束**：`P0.5` 阶段不新增 CLI Channel、不新增新响应原语、不新增新渠道，仅做当前实现的组织重构。 |

**第四章：编排引擎层 (Orchestration Engine) P0.5 规格详述**

| specId | 对应子模块 | spec 描述 |
| :---- | :---- | :---- |
| OE-P05-01 | 目录显性化 (Directory Layout) | **结构整理**：将基类、注册中心、运行时、上下文管理器、导出层等文件组织得更清晰，强化编排层骨架结构的可读性。 |
| OE-P05-02 | 协议边界保持稳定 (Stable Contracts) | **协议稳定**：保持 `BaseOrchestrator`、`CapabilityHub`、`RuntimeContext` 等现有协议语义不变，不在 `P0.5` 阶段引入新编排能力。 |
| OE-P05-03 | 运行时与骨架分层 (Runtime Separation) | **层次清晰**：将编排抽象、LangGraph 运行时占位实现、上下文维护逻辑的代码边界进一步显性化。 |
| OE-P05-04 | 注册中心组织优化 (Registry Layout) | **注册组织**：保留现有 Agent 注册中心能力不变，但优化其与导出层、调用代理之间的代码组织关系。 |
| OE-P05-05 | 上下文组件独立 (Context Components) | **上下文显性化**：让状态初始化、上下文更新、状态持久化相关组件在目录结构上更清晰，为后续 `P1` 增强做准备。 |
| OE-P05-06 | 对外依赖收束 (Upper Layer Dependency) | **依赖收束**：减少 `app` 层或其他层对编排层内部文件路径的直接耦合，优先通过稳定导出入口访问。 |
| OE-P05-07 | 无运行语义变更约束 (No Runtime Change) | **边界约束**：`P0.5` 阶段不要求完整落地 LangGraph 真正执行链，仅整理当前骨架与运行时代码组织。 |

**第五章：技能与工具层 (Skill Hub) P0.5 规格详述**

| specId | 对应子模块 | spec 描述 |
| :---- | :---- | :---- |
| SH-P05-01 | 核心框架目录 (Core Directory) | **框架核心独立**：将能力中心、协议、工具规范解析、导出与装配逻辑收敛到 Skill Hub 的核心目录中。 |
| SH-P05-02 | 安全原语目录 (Primitives Directory) | **原语独立**：将文件读写、命令执行、授权票据、策略判定等安全原语实现迁移到独立子目录。 |
| SH-P05-03 | 内置工具目录 (Builtin Tools Directory) | **工具独立**：将浏览器等现有内置工具实现迁移到独立子目录，与 Skill Hub 框架核心职责分离。 |
| SH-P05-04 | 导出接口兼容 (Export Compatibility) | **导出兼容**：在完成目录迁移后，尽量保持 `bootstrap`、`exports`、能力注册链路的现有行为与调用方式不变。 |
| SH-P05-05 | Tool Spec 路径清晰化 (Tool Spec Layout) | **规范显性化**：将 Tool Spec 解析器与工具实现的边界组织得更清晰，方便后续继续增强工具定义规范。 |
| SH-P05-06 | 能力路由保持稳定 (Stable Capability Routing) | **路由稳定**：保留现有 Capability Hub、模型能力注册、PyTool 注册的行为语义，不在 `P0.5` 阶段修改路由策略。 |
| SH-P05-07 | P0.5 重点层级 (Priority Refactor Layer) | **重点约束**：Skill Hub 是 `P0.5` 阶段最优先完成目录整理的层级之一，应作为六层内部组织优化的代表性样板。 |
